## 7. Issues / Decisions

### Vấn đề phát sinh

- **RED baseline tension.** Initial count had `test_navigate_falls_back_to_tile_click_when_spa_bounces` (B27 test) potentially needing removal because B29 raising fast would bypass the tile-click fallback. Resolved by placing B29's raise at the END of `navigate_to_edit` (replacing the existing "Failed to enter edit mode" raise, NOT adding a new early-raise), so the tile-click fallback stays reachable and B27's test still passes. Only effect on the pre-existing `test_navigate_raises_when_not_in_edit_mode` was a `match=` regex swap from `"edit mode"` → `"SPA stripped"`.
- **Exception propagation in `click_action_button`.** The existing `try/except Exception: continue` pattern would have swallowed B28's `RuntimeError`. Fix: added `except RuntimeError: raise` before the generic `except Exception: continue` in both Pass 1 and Pass 2. Unit test `test_click_action_button_raises_on_disabled` locks this in.

### Quyết định đã đưa (judgment calls)

- **One combined commit, not three.** B28, B29, B30 share a single root cause (INV-5 re-revision — extend mints NEW media_id), and the fix strategy is "resolve structurally at claim layer (B30); keep worker-layer diagnostics (B28/B29) as defensive backstops for future regressions." Splitting into three commits would have obscured the causal chain. Supervisor task spec also prescribed one combined commit.
- **Walk-up iteration bound: 16.** Picked as a large-enough-for-any-real-chain, small-enough-to-bound-bad-input value. Real chains observed to date are ≤ 4 levels; 16 gives 4× headroom. Pure safety net against malformed `parent_job_id` cycles.
- **B30 walks up for media_id + edit_url ONLY; profile + project_url still come from direct parent.** Rationale: profile is per-chain-invariant (INV-1) and project_url is also per-chain-invariant (all operations on the same project). Only media_id / edit_url vary per operation. Limiting the walk-up to the two fields that actually need it keeps the blast radius minimal and preserves the invariance-by-construction of the other two.
- **Walk-up terminates on `parent_job_id IS NULL`.** The root L1 has no parent; if we reach it while still on an extend-video (impossible by construction since L1 is always `text-to-video`, but defensive), we stop and inherit from it.
- **B28 error message cites `FLOW_BUTTON_EXACT §5.1`.** Per supervisor spec verbatim — this is a future-facing anchor even if §5.1 doesn't yet exist; creates a grepable pointer for the next operator who hits this diagnostic.

### Bug candidates phát hiện NHƯNG KHÔNG fix (out of scope)

- **INV-5 insert/remove preservation claim.** The 2026-04-16 FLOW_MULTILEVEL_JOBS §10 run concluded that `insert-object` and `remove-object` preserve media_id. This has NOT been empirically re-verified post-2026-04-19 (Tier 2 chains via insert/remove haven't completed to a child since B22 landed). SPEC §A.1 INV-5 matrix now explicitly marks both as **TBD**. If either empirically mints new media too, a B31 (similar walk-up for insert/remove parents) may be needed — but not fixing now without data.
- **FLOW_BUTTON_EXACT §5.1.** B28's error message references this anchor but the doc may not yet have a §5.1 describing extend-child lockout. A brief follow-up to add that section would make the diagnostic fully self-serving.
