### 7.2 Bug candidates (out of scope to fix — flag for supervisor)

> These are new empirical findings. Not retrying or attempting fix in this session.

1. **[B28-candidate, P0] Action sidebar missing on extend-output `/edit/` URL.**
   - Reproduce: run chain t2v → extend → any L2+ op (insert | camera | remove). The L3 op navigates to `/edit/{extend_output_media_id}` successfully (`Video element loaded`) but finds zero matches for action buttons (`Insert` title, `add_box` icon · `Camera` title, `videocam` icon).
   - Evidence (this session):
     - J3 insert-object on J2 extend output: `Video element loaded` → `RuntimeError: Failed to find Insert button`.
     - Diagnostic camera on J2: `Video element loaded` → `RuntimeError: Failed to find Camera button`.
   - Contrast: Tier 2 Run 10 (2026-04-19 earlier today) ran t2v → **camera-move** → insert and all three completed. The delta is the middle operation (`extend-video` in this session vs. `camera-move` in Run 10). Extend output's `/edit/` appears to render a different layout than t2v or camera output's `/edit/`.
   - Affects: any chain with `extend-video` in the middle is blocked.
   - Not investigated: DOM comparison of the two edit modes. Belongs in a follow-up Tier 1 DOM-probe session.

2. **[B29-candidate, P0] L1 edit URL breaks after a sibling extend-video runs.**
   - Reproduce: complete L1 t2v (media_A). Complete L2 extend-video off it (media_B = NEW uuid). Now navigate to `/edit/{media_A}` — video element does NOT load (15s timeout), action sidebar does NOT render; Camera button click via icon fallback "succeeds" (clicks something) but the preset grid is not visible.
   - Evidence: all 3 Test-4 cameras on J1 parent. Pattern:
     ```
     Navigating to edit URL: .../edit/6842325d-…  (J1's media_id)
     WARNING Video element not found after 15s — proceeding anyway
     INFO    Clicked mode button via icon='videocam' (requested title='Camera')
     ERROR   Could not click+verify camera preset: <direction>
     RuntimeError: Failed to find camera preset: <direction>
     ```
   - Likely root cause: Flow's SPA treats extend-video output as the new "current" media of the project; the previous media's `/edit/` URL becomes a stale-redirect or empty-state.
   - B27 relevance: `navigate_to_edit` was simplified to `page.goto(edit_url)` direct-fast-path (2026-04-19 commit `9519c06`). Pre-B27 would still have gone through the same code path here (project_url fallback only activates on rare 404) — B27 is probably not the cause.

3. **[INV-5 empirical contradiction] `extend-video` minted NEW `media_id`.**
   - SPEC §A.1 INV-5 behavior matrix (after `3d7b884` revision) states: "`extend-video` / `insert-object` / `remove-object` — Preserved (Flow updates in-place)." Evidence cited `FLOW_MULTILEVEL_JOBS.md §10` (2026-04-16) and Tier 2 Run 10 (2026-04-19).
   - This session: J1 `6842325d-8bdb-…` → J2 `extend-video` → J2's stored `media_id` = `1a6e3b77-d91a-…` (NEW uuid). Re-extracted from post-submit `page.url` via `finalize_operation`. Two separate `downloads\ext_720p_…mp4` files (see J2 row §5) indicate the download loop ran twice (original + upscale) — not a duplicate-clip issue.
   - Bug-candidates 1 and 2 **may be caused by** this empirical reality: if extend mints new media, the parent's `/edit/` URL becoming stale (B29) and the extend-output's `/edit/` having a different layout (B28) are symptoms of the change, not independent bugs.
   - Action: SPEC INV-5 matrix needs revision to reflect that `extend-video` **mints new media_id**, matching `camera-move`. Evidence from Tier 2 Run 10 should be re-examined: Run 10 did NOT include `extend-video`, so the "preserved" claim for extend rests only on `FLOW_MULTILEVEL_JOBS.md §10` (3 days older, different account). Belongs in a separate SPEC-revision task.
