## §1 — What / Why

### B20 — three residual fuzzy `'Veo'` selectors

B20 was originally logged in the B19 follow-up list (SPEC §D.4 B19) as a P2
candidate: the pre-B26 `flow/model_selector.py` used `button:has-text('Video')`
to open the model dropdown; on the current Flow DOM that selector also matched
the aspect chip. B26 (commit `d4fca1a`, 2026-04-19) incidentally absorbed the
primary `'Video'` collision by rewriting `chip_selectors` to
`button[aria-haspopup='menu']` + exact crop-icon / `arrow_drop_down` ligatures.

Three fuzzy `'Veo'` sites survived B26's rewrite:

| # | Line | Pre-fix selector | Context |
|---|---|---|---|
| 1 | 281 | `page.locator("button").filter(has_text="Veo")` | `_open_model_dropdown` — find in-panel model name button |
| 2 | 556 | `"button:has-text('Veo')"` | `get_current_model` — read chip text |
| 3 | 557 | `"[role='button']:has-text('Veo')"` | `get_current_model` — ARIA fallback |

Same bug class as B20-primary: bare substring match on "Veo" can leak to any
visible button whose textContent happens to include "Veo" — for example a
video-library label reflecting a previously selected model in an unrelated
panel, or a tooltip mirrored via `aria-describedby`.

### B21 — stray `print()` in model_selector.py

B21 was P3 in the B19 follow-up list: "worker emits a stray bare
`arrow_drop_down` line to stdout during model selection". At master
`83f621f`, `grep "print(" flow/model_selector.py` returns zero hits — the
stray `print()` is not present in the current code. Most likely removed
incidentally during an earlier B17 (LP pre-check, commit `f5dab42`) or B26
(commit `d4fca1a`) refactor without being called out in a changelog. No
source artefact remains to fix.

## §2 — Canonical pattern (reused, not invented)

Per `docs/FLOW_BUTTON_EXACT.md §1.6` and the B26 `chip_selectors` precedent
(`flow/model_selector.py:65-70`), the locale-independent anchor for the model
chip is:

```
button[aria-haspopup='menu']:has(i:text-is('arrow_drop_down'))
```

When the match must be narrowed to the Veo variant (either the in-panel
"Veo 3.1 — Fast ▼" button OR the chip reading the current model), combine
with:

```python
.filter(has_text=re.compile(r"^Veo", re.IGNORECASE))
```

`re.compile(r"^Veo", re.IGNORECASE)` anchors on textContent start and is a
principled Python-side regex filter — NOT a fragile Playwright CSS
substring match. The trip-wire forbids the fragile forms but permits the
regex form.

## §3 — Diff shape

### `flow/model_selector.py` site 1 (`_open_model_dropdown`, pre-fix line 281)

Before:

```python
veo_btns = page.locator("button").filter(has_text="Veo")
```

After:

```python
veo_btns = page.locator(
    "button[aria-haspopup='menu']:has(i:text-is('arrow_drop_down'))"
).filter(has_text=re.compile(r"^Veo", re.IGNORECASE))
```

Two-pass iteration logic downstream (non-LP first, LP fallback) untouched —
only the locator narrowing changed.

### `flow/model_selector.py` sites 2+3 (`get_current_model`, pre-fix lines 556-557)

Before (2-entry list with fallback loop):

```python
for sel in [
    "button:has-text('Veo')",
    "[role='button']:has-text('Veo')",
]:
    try:
        el = page.locator(sel).first
        if await el.is_visible(timeout=1000):
            return (await el.inner_text()).strip()
    except Exception:
        continue
```

After (single canonical anchor + regex filter):

```python
chip = page.locator(
    "button[aria-haspopup='menu']:has(i:text-is('arrow_drop_down'))"
).filter(has_text=re.compile(r"^Veo", re.IGNORECASE)).first
if await chip.is_visible(timeout=1000):
    return (await chip.inner_text()).strip()
```

The `[role='button']` ARIA fallback was semantically redundant — a `<button>`
element already has implicit `role="button"`, and the `aria-haspopup='menu'`
gate is the real discriminator. Collapsing 2 entries → 1 removes dead paths.

### `tests/test_model_selector.py` (+1 new test, 53 lines)

New `test_no_fuzzy_veo_selector` — source-level trip-wire. Forbidden patterns:

- CSS: `has-text('Veo')` and `has-text("Veo")`
- Raw filter: `filter(has_text="Veo")` and `filter(has_text='Veo')`

Regex-based filters (`re.compile(...)`) remain permitted.

## §6 — Why B21 was not re-probed in production

The original B21 report (B19 follow-ups) noted the stray `print()` emitted
`arrow_drop_down` to stdout. At `master 83f621f` the only `arrow_drop_down`
references in `flow/model_selector.py` are:

- String literals inside CSS selectors (`i:text-is('arrow_drop_down')`)
- String literals inside JS fallback closures (`lower.includes('arrow_drop_down')`)
- `_has_dropdown_arrow` helper comment
- This report's own cross-reference

None use `print()`. `grep "print(" flow/model_selector.py` → 0 hits
confirms no stdout leakage path exists.

Adding a source-level trip-wire against `print(` across the whole `flow/`
package was considered and rejected: `flow/` is expected to use
`logging.getLogger(__name__)` throughout, and there are other files
(e.g. auxiliary debug scripts) where `print` may be legitimate. A
file-scoped trip-wire just for `flow/model_selector.py` would solve B21
specifically but has negative ROI — the forbidden construct is already
absent, and regressing it would require a deliberate code change that a
reviewer would flag on the diff.

Decision: close B21 with no test, reference this report's §6 rationale
in SPEC.md and WORKPLAN.md.

| Collapse `_open_model_dropdown` two-pass (non-LP then LP) into one pass | Out of scope for B20/B21 — the two-pass is an LP-priority invariant (B8 lineage), not a fuzzy-selector issue. |

  trip-wire is the mechanical guarantee until then.

  master `83f621f`. It may be dead code. Keep for now (observability value
  if re-wired into a future diagnostics path) but mark for deletion review
