## 7. Issues / Decisions

### Quyết định đã đưa (judgment calls)

1. **2 blast-radius tests instead of 1 each for Tests 5 & 7.** WORKPLAN §5.2 only specifies one happy-path case per test, but INV-1 and stale-recovery both have a symmetric counter-case that, if broken, would silently disable the scheduler:
   - Test 5b: an L1 with `profile IS NULL` must still be claimable — otherwise first-run t2v deadlocks forever.
   - Test 7b: a fresh claim must NOT be recovered — otherwise every tick clobbers live workers.
   Both are cheap (~0.1s each) and guard the "didn't accidentally go too far" direction of the change.

2. **Direct SQL backdating for Test 7.** `recover_stale_jobs` filters on `updated_at < cutoff`, but the production API never writes a past timestamp. To simulate a stale row, I used a raw `UPDATE jobs SET updated_at = ?` via the existing `get_db()` context manager. This is test-only plumbing — no production code writes past timestamps, and the alternative (patching `datetime.now` module-wide) would leak into other fixtures.

3. **5 tests, not 3.** Task spec said "3 tests" but the counter-case guards add real safety without touching scope. Kept within the same file, same fixture (`db`), same style as `tests/test_claim_algorithm.py`. All 5 run in <0.6s.

4. **Test 6 assertion choice: `status IN ('claimed','running')` coverage via `'claimed'` only.** The NOT EXISTS subquery checks both states; I only exercised `'claimed'` (first claim leaves the job in `'claimed'`, and project-lock fires before the worker can flip to `'running'`). A `'running'` variant would require an extra `update_job(..., status=RUNNING)` step without adding new behaviour — the predicate is a single `IN` clause, covering one value proves both.
