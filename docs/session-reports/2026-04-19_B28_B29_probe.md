## 8. Fix direction

### 8.1 B28 — extend-output sidebar disabled

**Root cause:** Flow disables Insert/Remove/Camera on the extended-child clip. Matches the pre-existing `FLOW_BUTTON_EXACT.md §5.1 "Extend-child clip lockout"` gotcha that had been noted but never implemented as a workaround. User must select the PARENT clip in the timeline strip (scroll back + click original thumbnail) to re-enable non-extend modes.

**Concrete selector — the button IS there, just disabled:**
```
button[title='Camera'][disabled]     # present on extend-output /edit/
button[title='Insert'][disabled]
button[title='Remove'][disabled]
```

**Two mutually compatible fixes (scope-able as B28a + B28b):**

1. **Diagnostic fix (small, ships now):** change `click_action_button` in `flow/operations/_base.py:141` to call `is_enabled()` in addition to `is_visible()`, and emit a clearer error:
   ```python
   if await btn.is_visible(timeout=1500):
       if not await btn.is_enabled():
           logger.warning(
               "Mode button %r is present but disabled — likely extended-child "
               "lockout (FLOW_BUTTON_EXACT §5.1). Parent-clip select not implemented.",
               text,
           )
           continue  # try next candidate; all will fail the same way, then caller raises
       await btn.click(timeout=timeout_ms)
   ```
   Raise becomes `RuntimeError("Camera button disabled on this clip — extended-child lockout")` instead of the misleading `"Failed to find Camera button"`.

2. **Architectural fix (larger — B28 primary):** before clicking Insert/Remove/Camera on a job whose parent type is `extend-video`, navigate to the PARENT media's edit URL instead of the extend-output's. Implementation options:
   - In B22 claim-time inheritance: for `extend-video` parents, propagate the **parent's parent's** `media_id` (= original L1 media) to the child job, not the extend-output media. Subsequent Insert/Remove/Camera jobs then land on the enabled sidebar naturally.
   - Or, inside `navigate_to_edit`: after landing on `/edit/{extend_output_media}`, detect the disabled-sidebar state and re-navigate to the extend's parent.

   Preferred: the B22 inheritance path, because it keeps `navigate_to_edit` simple and locates the policy decision at the queue layer where parent-chain context is already available.

### 8.2 B29 — stale L1 URL after sibling extend

**Root cause:** Flow SPA treats the most recent media as the project's "current" media; `page.goto(/edit/{old_media_id})` drops the `/edit/` segment and the SPA lands on the library grid. Engine's tile-click fallback then clicks the wrong `videocam` icon (library "View videos") because it matches the same icon ligature.

**Immediate fix:** fail fast on URL stripping. In `flow/operations/_base.py::navigate_to_edit` after `await page.goto(target_url, …)` → `await asyncio.sleep(3)`, add:
```python
if "/edit/" not in page.url and target_url != page.url:
    raise RuntimeError(
        f"SPA stripped /edit/ segment on goto — likely stale media_id "
        f"(sibling extend moved the project's current media). "
        f"landed={page.url[:100]}  target={target_url[:100]}"
    )
```
This removes the misleading `"Video element not found after 15s — proceeding anyway"` warning path and the false-positive `videocam`-icon click. Callers (`run_camera`/`run_insert`/`run_remove`) will see a clear exception instead of chasing a broken state.

**Architectural fix:** tied to B28's B22 inheritance fix — if extend-video propagates the right media_id to siblings, the L1 parent's /edit/ URL is never navigated post-extend. This eliminates the B29 class entirely for chain workflows. Independent B29 fix only matters if a user/operator submits jobs with explicit `parent_job_id=<L1>` after a sibling extend has run.

### 8.3 Decision — document scope

- **Neither finding is "Flow doesn't render — unsupported, document scope".** Flow DOES render; the engine's interaction model is stale.
- B28 ✅ has a concrete selector (`button[title='Camera'][disabled]`) and two fix paths.
- B29 ✅ has a concrete detection (`"/edit/" not in page.url` post-goto) and two fix paths.
- Both bugs are P0 for chains where `extend-video` is in the middle.
